Base URLs
Production: https://us-central1-inink-c76d3.cloudfunctions.net/api
Local Emulator: http://127.0.0.1:5001/inink-c76d3/us-central1/api

Authentication
The API uses two authentication mechanisms.

Admin Secret (for /admin/* endpoints)
Used by Shopify app to manage merchants. No login step required.

Header	Value
X-Admin-Secret	<admin_secret>
Alternative (fallback): Authorization: Bearer <admin_secret>

Merchant API Key (for /api/* endpoints)
Issued once when a merchant is created via POST /admin/merchants. The backend hashes the key with SHA-256, resolves the shop_id, and enforces tenant isolation.

Header	Value
Authorization	Bearer <api_key>
API key format: ink_live_<48 hex characters>
Tenant Isolation: Every /api/* request is scoped to the merchant identified by the API key. Merchant A cannot read or modify Merchant B's data.

POST /admin/merchants
Create a new merchant and generate an API key. Called by Shopify app during the OAuth install callback.
NEW in v1.2.0: When a new merchant is created (not a reinstall), the system automatically initializes their sticker inventory with the default amount .
NEW in v1.3.0: Now supports both Shopify and custom (non-Shopify) merchants via the platform field. Added support for non-Shopify merchants with the platform parameter ("shopify" or "custom").
The api_key is returned exactly once in the response. It is stored only as a SHA-256 hash and cannot be retrieved again.
Request
POST /admin/merchants
Content-Type: application/json
X-Admin-Secret: <admin_secret>
Request Parameters
Field	Type	Required	Description
shop_domain	string	Yes	Shopify store domain, unique identifier
shop_name	string	Yes	Display name of the store
owner_email	string	Yes	Store owner's email address
platform	string	No	"shopify" or "custom" - Platform type (default: "shopify"). For Shopify merchants, shop_domain is required. For custom merchants, shop_domain is optional.
shopify_plan	string	No	Current Shopify plan tier (basic, shopify, advanced, plus)
merchant_category	string	No	Product category (sneakers, electronics, fashion)
merchant_region	string	No	Geographic region (US-East, EU-West)
avg_order_value	number	No	Average order value in USD
primary_carriers	array	No	List of primary shipping carriers
monthly_order_volume	integer	No	Approximate monthly order count
pre_ink_dispute_rate	number	No	Dispute rate before INK (0.0 to 1.0)

Response 201 Created
Field	Type	Required	Description
shop_id	string	Yes	Unique merchant identifier (Firestore document ID)
api_key	string	Yes	Merchant API key (ink_live_*). Only returned once.
is_new_install	boolean	Yes	true if new merchant, false if reinstalled
inventory_initialized	boolean	Yes	true if inventory was initialized for new merchant
initial_inventory_count	number	No	Number of stickers added (only if inventory_initialized is true)

Typical Workflow

Step 1: Merchant Installation (one-time)
•	Shopify merchant installs INK app
•	App calls POST /admin/merchants
•	System returns API key and initializes inventory
•	Merchant stores API key securely

Step 2: Package Enrollment (per shipment)
•	Merchant prepares package for shipment
•	Scans NFC sticker and calls POST /api/enroll
•	System deducts 1 sticker from inventory
•	Returns proof_id for tracking

Step 3: Media Upload (per shipment)
•	Merchant takes package photo
•	Calls POST /api/media/upload with proof_id
•	System stores media in Cloud Storage

Step 4: Mark as Delivered (when shipped)
•	Carrier delivers package
•	System calls PATCH /api/proofs/:proof_id/delivered
•	Records delivery timestamp and location
•	Calculates interaction window

Step 5: Customer Verification (on delivery)
•	Customer taps NFC sticker with phone
•	App calls POST /verify
•	System checks if within expected window
•	Records first tap timestamp


Step 6: Proof Retrieval (on demand)
•	Merchant or customer calls GET /api/proofs/{nfc_token}
•	System returns all proof details and media

Step 7: Inventory Management (as needed)
•	Merchant checks inventory with GET /api/inventory
•	Admin adjusts inventory with POST /admin/merchants/:shop_id/inventory


Merchant Authentication & Non-Shopify Support APIs (v1.3.0)
This section documents the new merchant authentication and non-Shopify support APIs added in v1.3.0.
POST /admin/users (New)
Create a merchant user account (email + password) linked to a specific merchant.
Path: /admin/users
Auth: X-Admin-Secret: <ADMIN_SECRET>
Request Parameters
Field	Type	Required	Description
merchant_id	string	Yes	shop_id of the merchant
name	string	Yes	User's display name
email	string	Yes	Login email (case-insensitive, globally unique)
password	string	Yes	Plaintext password (min length: 8 characters)


Response
Field	Type	Description
user_id	string	ID of the user document in merchant_users
merchant_id	string	Linked merchant (shop_id)
name	string	User's display name
email	string	User's login email (normalized to lowercase)
created_at	string	ISO timestamp when the user was created
POST /auth/login (New)
Merchant user login using email and password. Returns a JWT access token.
Path: /auth/login
Auth: None (public endpoint, credentials in body)
Request Parameters
Field	Type	Required	Description
email	string	Yes	User's login email
password	string	Yes	User's password
Response
Field	Type	Description
token	string	JWT access token (expires in 7 days by default)
user	object	User information (user_id, merchant_id, name, email, role)


GET /auth/validate (New)
Validate a JWT token and return the associated merchant and user info.
Path: /auth/validate
Auth: Authorization: Bearer <token>
Response (valid token)
Field	Type	Description
valid	boolean	true if token is valid
shop_id	string	Merchant ID (merchant_id in token)
user_id	string	User ID (user_id in token)
email	string	Email from token
role	string	Role from token (currently "merchant")
Response (invalid/expired/missing token): Always returns HTTP 200 with { "valid": false }

NEW: Delivery Outcome + Dispute APIs
PATCH /api/proofs/{proof_id}/outcome
Update delivery_outcome on a proof and store audit metadata (outcome_source, outcome_updated_at).
Request
PATCH /api/proofs/{proof_id}/outcome
Content-Type: application/json
Authorization: Bearer <api_key>
Request Body
Field	Type	Required	Description
delivery_outcome	string	Yes	One of: UNCONFIRMED | ACCEPTED | DISPUTED | EXPIRED | RETURNED
outcome_source	string	No	One of: system | merchant | webhook (defaults to merchant if omitted)

Response 200 OK
Field	Type	Description
proof_id	string	Proof ID
delivery_outcome	string	Updated outcome
outcome_source	string	Stored source
outcome_updated_at	string (ISO 8601)	Update timestamp

POST /api/proofs/{order_id}/dispute
Ingest dispute data (typically forwarded from Shopify webhooks) and store it on the proof record identified by order_id.
Request
POST /api/proofs/{order_id}/dispute
Content-Type: application/json
Authorization: Bearer <api_key>
Request Body
Field	Type	Required	Description
dispute_type	string	No	e.g. chargeback, inquiry, etc.
dispute_source	string	No	e.g. shopify
dispute_amount	string | number	No	Stored as string in DB
dispute_reason_code	string	No	Reason code
dispute_reason	string	No	Free-form reason text
dispute_outcome	string	No	e.g. open, won, lost
ink_evidence_submitted	boolean	No	Whether INK evidence was submitted

Response 200 OK
Field	Type	Description
proof_id	string	Resolved proof ID
order_id	string	Order ID
delivery_outcome	string	Always DISPUTED after this call
outcome_source	string	Always webhook after this call
outcome_updated_at	string (ISO 8601)	Update timestamp
dispute_updated_at	string (ISO 8601)	Dispute ingestion timestamp
